# Учёт заказов — онлайн-версия (сервер)

Это фундамент онлайн-версии программы: сервер на Node.js, вход по логину/паролю,
API для заказов и прайса, заготовка синхронизации с Kaspi API по 3 магазинам.

**Важно понимать сразу:** это первая, базовая версия. Из офлайн-файла сюда ещё
не перенесены фото-лайтбокс, колонка "дней в пути", красивая витрина для клиента,
этикетки со штрихкодом, импорт/экспорт Excel — это следующие шаги (см. `CHANGELOG.md`).

## Как запустить локально (проверить, что всё работает)

Нужен установленный Node.js версии 18 или новее.

```bash
cd server
npm install
cp .env.example .env
# откройте .env и при желании поменяйте ADMIN_USER/ADMIN_PASS, SESSION_SECRET
npm start
```

Откройте `http://localhost:3000` — увидите экран входа (по умолчанию `admin` / `admin`,
задаётся через `.env`).

## Где разместить (хостинг)

Коротко: рекомендую **Render.com** как самый простой старт. Ниже — почему и как.

### Важная техническая деталь про хранение данных

Программа хранит все данные (заказы, прайс, фото) в одном файле `data.sqlite`
и папке `uploads/` рядом с кодом. Это просто и без лишних настроек, но у этого
есть последствие: **на большинстве хостингов файловая система "одноразовая"** —
при каждом обновлении/перезапуске сервера она обнуляется, и данные пропадут,
если не подключить постоянный диск (persistent disk / volume).

Поэтому при развёртывании обязательно нужно:
1. Подключить постоянный диск на хостинге (платная опция почти everywhere)
2. Указать в переменных окружения `DB_PATH` и `UPLOAD_DIR`, чтобы они указывали
   на этот диск (например `/data/data.sqlite` и `/data/uploads`)

Без этого шага — можно потерять все заказы при следующем обновлении версии.

### Вариант 1 — Render.com (рекомендую для старта)

1. Зарегистрируйтесь на render.com, подключите GitHub-репозиторий с этим кодом
2. Создайте **Web Service**, Node.js определится автоматически
   - Build command: `npm install`
   - Start command: `npm start`
3. На бесплатном тарифе сервис "засыпает" после 15 минут простоя (первый
   заход после этого — 30-60 секунд ожидания). Если это неудобно для
   ежедневной работы — платный тариф от ~$7/мес держит сервис постоянно включённым.
4. Добавьте **Persistent Disk** (платно, обычно от $1 за GB/мес) — например,
   на 1GB, точка монтирования `/data`
5. В переменных окружения (Environment) задайте:
   - `SESSION_SECRET` — длинная случайная строка
   - `ADMIN_USER`, `ADMIN_PASS` — свои значения
   - `DB_PATH=/data/data.sqlite`
   - `UPLOAD_DIR=/data/uploads`
   - при готовности — `KASPI_TOKEN_SHOP1/2/3` и `KASPI_SHOP1/2/3_NAME`
6. Деплой — Render сам даст вам адрес вида `https://ваш-проект.onrender.com`

### Вариант 2 — Railway.app

Railway начиная с 2024 года без бесплатного тарифа (нужна оплата от ~$5/мес),
зато обычно чуть быстрее в развёртывании и с приятным интерфейсом. Тоже
поддерживает постоянные тома (volumes) — логика та же: подключить том,
указать `DB_PATH`/`UPLOAD_DIR` на путь внутри него.

### Что выбрать

Если хочется сначала бесплатно попробовать в деле — Render. Если не смущает
небольшая ежемесячная плата и важна скорость/удобство — Railway. Для 3
магазинов и одного администратора нагрузка минимальная, хватит самого
дешёвого платного тарифа любого из них.

## Обновление версий (как договорились — чётко и без сюрпризов)

1. В `package.json` поле `"version"` — бампаем при каждом значимом обновлении
   (`0.1.0` → `0.2.0` и т.д.)
2. В `CHANGELOG.md` — новая запись сверху с тем, что изменилось
3. Версия видна пользователю в подвале страницы и через `GET /api/meta/version`
4. Деплой обновления = обычный `git push` в подключённый репозиторий — хостинг
   сам пересоберёт и перезапустит (данные не потеряются, если постоянный диск
   подключён как описано выше)

## Структура проекта

```
server/
  server.js          — точка входа, настройка Express
  db.js              — работа с данными (SQLite через встроенный node:sqlite)
  kaspi.js           — интеграция с API Kaspi (см. предупреждение ниже)
  middleware/auth.js — проверка авторизации
  routes/
    auth.js          — вход/выход/смена пароля
    orders.js        — CRUD заказов
    price.js         — CRUD прайса + категории
    kaspi.js          — синхронизация заказов с Kaspi
    meta.js          — версия приложения
  public/index.html  — фронтенд (простой, будет дорабатываться)
  uploads/           — загруженные фото (локально; на хостинге — на диске)
```

## О синхронизации с Kaspi — важная оговорка

Модуль `kaspi.js` написан по официальной документации Kaspi
(`https://guide.kaspi.kz/partner/ru/shop/api/orders/`), но **не протестирован
на реальном токене** — у меня не было доступа к API Kaspi при разработке.

Что нужно сделать при первом подключении:
1. Получить токен API в кабинете продавца каждого магазина (раздел API)
2. Прописать в `.env`: `KASPI_TOKEN_SHOP1=...`, `KASPI_SHOP1_NAME=Название магазина`
   (и так же для SHOP2, SHOP3)
3. Нажать "Синхронизировать" в разделе Kaspi в приложении
4. Если Kaspi вернёт данные в структуре, отличной от ожидаемой в
   `mapKaspiOrder()` (в `kaspi.js`) — пришлите мне пример ответа (можно
   через Postman/браузер), и я поправлю маппинг полей под реальный формат

Учитывайте также, что Kaspi отдаёт заказы постранично — сейчас забирается
только первая страница (100 заказов). Если заказов больше — понадобится
доработать пагинацию, это можно сделать на следующем шаге.

## Про роли пользователей

Пока все, кто вошёл — администраторы (одна общая роль). В структуре
пользователя уже есть поле `role`, так что добавить разграничение прав
(например, "только просмотр" для сотрудников) можно будет позже без
переделки с нуля.
