# История версий

Правило на будущее: при каждом обновлении — новая запись сверху + бампаем
`"version"` в `package.json`. Версия видна в самом приложении (подвал страницы)
и через `GET /api/meta/version`.

## v1.0.0 — крупное обновление: статусы обработки, грузовые места, печать, роли, БД

**Это самое большое обновление на данный момент — реализовано по развёрнутому ТЗ.**
Ниже — сводка, подробности в отчёте, который прислан отдельным сообщением.

### Хранилище данных
- **Переход с lowdb (JSON-файл) на настоящую БД — SQLite**, через встроенный в сам
  Node.js модуль `node:sqlite` (без внешних пакетов, без проблем со сборкой)
- Формат файла БД теперь `data.sqlite` (был `data.json`) — переменная `DB_PATH`
  теперь должна указывать на `.sqlite`-файл, поправьте на хостинге при обновлении

### Статусная цепочка обработки заказов (для заказов Kaspi)
- Предзаказ → Упаковка → Этикетка распечатана → Готов к отгрузке → Отгружен,
  плюс «Отменён» с любого активного этапа
- Обычные переходы — только на следующий шаг. Перепрыгнуть несколько шагов
  (например сразу в «Отгружен») может только администратор и только с указанием причины
- Полная история изменений статуса — кто, когда, откуда куда, по какой причине

**Важно:** статус обработки Kaspi-заказа — это отдельное поле от статуса «Прихода»
(в пути/пришло/проблема). Раньше по ошибке чуть не свёл их в одно поле — в итоге
развёл: `status` (обработка Kaspi-заказа) и `receiveStatus` (Приход) — это два
разных процесса и они не должны были смешиваться

### Грузовые места
- К заказу можно добавить одно или несколько грузовых мест, у каждого — отметка
  «Сформировано» и «Этикетка распечатана»
- Перевод в статус «Готов к отгрузке» заблокирован, пока не сформировано
  ни одно грузовое место

### Контролируемая печать
- Печать требует явного подтверждения «Подтверждаю, что документ распечатан» —
  само открытие окна печати статус не меняет
- При повторной печати — предупреждение (когда и кем печаталось в прошлый раз)
  и обязательная причина повтора
- Первая успешная печать на стадии «Упаковка» автоматически переводит заказ
  в статус «Этикетка распечатана»

### Роли пользователей
- Администратор — полный доступ, включая журнал, исправление статусов, управление
  пользователями
- Склад — заказы, упаковка, печать, грузовые места, обычные переходы статуса,
  без доступа к журналу и настройкам

### Журнал действий
- Отдельная вкладка «Журнал» (только для админа) — кто, когда, что сделал,
  было/стало, комментарий

### Безопасность
- Простая защита от подбора пароля (блокировка на 5 минут после 5 неверных попыток)
- Проверка типа и размера загружаемых фото на сервере

## v0.9.0 — печать накладной (грузового места) и понятные детали заказа
- **Ссылка на накладную — из данных, которые уже присылает Kaspi** (`kaspiDelivery.waybill`):

  этого сырые данные от Kaspi (для сверки)

## v0.8.0 — фильтры как в кабинете Kaspi, цветные статусы, печать списка
- **Единый фильтр вместо двух** — стадии как в кабинете Kaspi: Предзаказ / Упаковка /
  Передача / Передано на доставку / Завершён / Отменён / Возврат / Архив. Это лучшее
  приближение по доступным полям (`assembled`, `courierTransmissionDate`, `preOrder`,
  `status`, `state`) — **не гарантированно 1-в-1** с тем, как это устроено внутри
  кабинета Kaspi, но должно быть намного ближе и понятнее прежнего. Если на конкретном
  заказе стадия определится неверно — пришлите его код, поправим логику именно под него
- **Статус теперь цветной** — как светофор: зелёный (доставлено/завершено), жёлтый/оранжевый
  (в процессе), красный (отменён), фиолетовый (предзаказ), серый (архив)
- **Печать списка** — кнопка «🖨️ Печать списка» печатает то, что видно с текущим фильтром
  (например только «Упаковка») — номер, код Kaspi, наименование, магазин, стадия
- Про «Возвраты» отдельно: похоже, это в Kaspi отдельный от заказов раздел (со своими
  подстатусами: новые заявки/на доставке/ожидают решения/споры) — вероятно отдельный API,
  не часть `/orders`. Пока показываем то, что видно по статусу самого заказа
  (RETURN_REQUESTED/RETURNED); полноценная интеграция с разделом возвратов — это
  отдельная задача, если понадобится

## v0.7.1 — вероятная причина "не все активные заказы видно"
- **Похоже, нашли корень проблемы**: синхронизация забирала только первые 100 заказов
  (одна страница ответа Kaspi). Если у магазина за 14 дней больше 100 заказов —
  самые свежие (те, что прямо сейчас "на упаковке") могли просто не попасть в эту
  первую сотню. Теперь синхронизация проходит **все страницы** ответа, а не только первую
- Названия товаров теперь тянутся только для неархивных заказов (чтобы не делать
  лишние сотни запросов на историю, которая и так не нужна по умолчанию)
- Нужно проверить: нажмите «Синхронизировать сейчас» ещё раз и сравните счётчики
  с личным кабинетом Kaspi — если расхождение осталось, разбираемся дальше

## v0.7.0 — архив по запросу, фильтр по предзаказу
- **Архивные заказы больше не сохраняются автоматически** при обычной синхронизации —
  Kaspi и так их хранит у себя, незачем дублировать. Если нужно посмотреть архив —
  галочка «включая архив» рядом с кнопкой синхронизации подтянет его тоже
- **Второй фильтр — «Предзаказ»**: работает независимо от фильтра по статусу и
  комбинируется с ним (например: статус "Принят продавцом" + "Только предзаказ")
- В результате синхронизации теперь видно, сколько архивных заказов было пропущено

## v0.6.0 — вкладка «Отмены и возвраты»
- Новая вкладка собирает заказы со статусами CANCELLED / CANCELLING / RETURN_REQUESTED / RETURNED
- По каждому — кнопка «Обработать»: можно оставить заметку и отметить «Закрыто (решено)»
- Закрытые по умолчанию скрыты (галочка «Показывать закрытые», чтобы посмотреть историю)
- Важная оговорка: количество заказов в приложении и в личном кабинете Kaspi может не
  совпадать — API отдаёт только заказы за последние 14 дней по дате создания (жёсткое
  ограничение самого Kaspi), а кабинет показывает все активные без такого ограничения.
  Также вкладки кабинета («Упаковка», «Передача» и т.п.) — это смесь нескольких полей
  сразу, а не один статус, так что зеркально повторить их пока не пытаемся

## v0.5.0 — реальные поля Kaspi, наименование товара, удобнее меню
- **Статус исправлен по-настоящему**: оказалось, я перепутал два поля Kaspi — `state`
  (логистическая стадия: KASPI_DELIVERY/ARCHIVE) и `status` (настоящий статус заказа:
  ACCEPTED_BY_MERCHANT, COMPLETED, CANCELLED и т.д.). Теперь "Статус" показывает `status`,
  а отдельная колонка "Доставка" — `state`
- **Наименование товара** — в самом заказе Kaspi его не передаёт, нужен отдельный запрос
  к позициям заказа. Добавлен (`fetchOrderEntries`) — тянется при синхронизации небольшими
  партиями, чтобы не перегружать API. Путь запроса пока не проверен на реальных данных —
  если наименование всё ещё не появится после этого обновления, посмотрим через «Детали»
- **Фильтр по умолчанию** — теперь при заходе на вкладку «Заказы» сразу видно «Активные»
  (то есть не архивные), а не всё подряд с самого старого
- **Колонки** — спрятаны за кнопку «⚙️ Колонки», не занимают место на экране постоянно
- Лёгкое обновление внешнего вида меню — закруглённые кнопки-вкладки вместо плоских полосок

## v0.2.0 — Приход / Заказы (Kaspi) разделены
- Вкладка «Заказы» переименована в **«Приход»** — это ручной учёт (в пути/пришло/проблема),
  как и раньше. Каждой записи теперь автоматически присваивается уникальный номер (`receiptNumber`)
- Новая вкладка **«Заказы»** — сюда попадают только заказы, подтянутые из Kaspi по API.
  У каждого — свой уникальный номер (`displayNumber`), код заказа Kaspi, магазин-источник,
  наименование (если Kaspi его отдаёт) и дата передачи курьеру
- Токены API для 3 магазинов Kaspi теперь можно вводить **прямо в интерфейсе**
  (вкладка «Kaspi: настройки»), не только через переменные окружения на сервере.
  Сохранённый токен показывается в замаскированном виде (не как полноценное шифрование,
  а просто чтобы не светить его на экране лишний раз)
- Данные обоих типов заказов физически хранятся в одном списке с полем `source`
  (`manual` / `kaspi`) — так удобнее синхронизировать и фильтровать

## v0.1.0 — первая онлайн-версия (фундамент)
- Сервер на Node.js/Express, вход по логину/паролю (без ролей — задел под них есть в структуре пользователя)
- Хранилище данных — простой JSON-файл (`data.json`) через lowdb, без внешней БД
- API для заказов и прайса (CRUD)
- Загрузка фото (`/api/upload`), отдача через `/uploads/...`
- Заготовка синхронизации с Kaspi API (модуль `kaspi.js`, до 3 магазинов через переменные окружения) —
  **не проверено на реальном токене**, см. README
- Простой фронтенд-каркас (вход, список заказов/прайса, кнопка синхронизации Kaspi)
- Категории прайса — выпадающий список с возможностью добавить новую "на месте"
